### Ответ про cloudns: сам по себе — нет

cloudns — это **только авторитативный DNS** (отвечает на DNS-запросы, отдаёт наш
**реальный** IP). Он **не проксирует** трафик. Обход блокировок у Cloudflare даёт именно
**proxy (оранжевое облако)**: клиент коннектится на anycast-IP Cloudflare, а не на наш
режущийся IP. cloudns этого не умеет:

- Если блок по **IP/DPI** — cloudns всё равно вернёт тот же режущийся IP, клиент упрётся
  в него.
- Если блок по **DNS у Wi-Fi-провайдера** — смена нашего авторитативного DNS не помогает,
  провайдер подменяет резолв на своём резолвере (клиент даже не спрашивает cloudns).

**Вывод:** proxy от Cloudflare нужен в любом случае. Но домен покупать/переносить не надо —
достаточно **сменить неймсервера `fatklyuchi.space` с cloudns на Cloudflare** (free-план
Cloudflare требует полной делегации NS; partial/CNAME-setup — только на Business). То есть
cloudns для этого домена заменяется на Cloudflare, а не дополняет его.

⚠️ Смена NS уводит в Cloudflare **весь DNS** `fatklyuchi.space`. Перед переключением снять
**полный список текущих записей в cloudns** и перенести их в Cloudflare, иначе отвалится то,
что на этом домене висит (панель/whitelist, если они на нём же).

### Что это меняет в плане

## Проблема (со слов Robert)

Страница подписки переадресована на сервер с «белыми списками». Из-за этого ключи
вставляются и обновляются **только с мобильного интернета**, а на Wi-Fi — нет.
Robert предложил «сделать переадресацию через 2 сервера».

## Диагноз

Симптом «работает на мобильном, не работает на Wi-Fi» — это про **доступность домена
подписки**, а не про сам VPN. Почти всегда одно из двух:

1. **DNS-фильтрация Wi-Fi-провайдера** — домашний/офисный/публичный роутер отдаёт свой
   DNS, который для домена подписки (или сервера, куда идёт редирект) возвращает
   пусто/заглушку. У мобильного оператора этой блокировки нет.
2. **Блокировка по IP / DPI** — IP сервера подписки в чёрных списках у проводных
   провайдеров; мобильные операторы его не режут.

«Сервер с белыми списками» = скорее всего тот сервер, на который **редиректит** страница
подписки. Его домен/IP и режется на Wi-Fi.

## Решение — Вариант А (Cloudflare)

Заводим `fatvdsnvv.space` в Cloudflare. Клиенты ходят на IP Cloudflare (не режут ни на
Wi-Fi, ни у проводных провайдеров), Cloudflare сам тянет конфиг с исходного сервера.
Заодно бесплатный HTTPS и скрытый реальный IP. Идея Robert «через 2 сервера» — это и есть
reverse-proxy; Cloudflare делает это без поднятия своего сервера.

Альтернатива (Вариант B, если Cloudflare почему-то нельзя) — свой nginx-reverse-proxy на
отдельном общедоступном сервере, A-запись поддомена → его IP, форвардит на whitelist-сервер.

## Что нужно от Robert

1. **IP сервера подписки** — машина, на которую сейчас редиректит страница подписки
   (сервер с белыми списками).
2. **Доступ к домену `fatvdsnvv.space`** — регистратор/хостинг + вход в личный кабинет
   (сменить неймсервера + снять полный список текущих DNS-записей).
3. **Cloudflare** — бесплатный аккаунт на cloudflare.com (Robert заводит и даёт доступ,
   либо заводим мы).

## Что делаем мы

1. Добавляем домен в Cloudflare → CF выдаёт 2 своих неймсервера (`xxx.ns.cloudflare.com`).
2. **Переносим все текущие DNS-записи** домена в Cloudflare (почта, другие поддомены — что
   есть), чтобы при переключении ничего не отвалилось.
3. A-запись поддомена подписки → IP сервера, режим **Proxied** (оранжевое облако).
4. HTTPS (Full/Flexible), проверяем открытие подписки на «плохом» Wi-Fi.

## Единственное действие Robert у регистратора

Сменить неймсервера домена на 2, что выдаст Cloudflare. После этого весь DNS домена
управляется из Cloudflare — остальное делаем мы.

⚠️ Перевод на неймсервера Cloudflare = **весь DNS домена**, не только подписка. Поэтому
до переключения снять полный список текущих записей (п.2 доступов) и перенести их.
