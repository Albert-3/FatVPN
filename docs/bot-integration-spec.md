# ТЗ: интеграция Telegram-бота с BFF FatVPN (черновик)

> Заготовка для Дня 6. Адресовано разработчику Telegram-бота заказчика.
> Источник контекста: раздел 3 `VPN-App-Project.md`.

## Задача

Кнопка **«Сменить ключ»** в боте должна выдавать ключ в новом формате
(короткий токен) вместо длинной sub-ссылки Remnawave, сохраняя текущий
`expires_at` пользователя.

## Что НЕ меняется

- Логика продажи/оплаты — не трогаем.
- Боевой бот работает как есть до momента переноса (см. план миграции ниже).
- Работа ведётся только на тестовом боте `@testfatvpnnbot`.

## Что нужно изменить в боте

При нажатии «Сменить ключ»:

1. Бот, как и раньше, находит текущую подписку пользователя в Remnawave
   (по существующей логике) и получает её `expires_at`.
2. Вместо возврата пользователю прямой sub-ссылки, бот:
   - генерирует короткий токен (формат — уточнить, например 8 символов,
     `[A-Z0-9]`);
   - вызывает `POST /internal/tokens` на BFF FatVPN (см. `api-contract.md`):
     ```json
     {
       "shortToken": "AB12CD34",
       "remnawaveSubscriptionId": "<id подписки в Remnawave>",
       "expiresAt": "<текущий expires_at пользователя>"
     }
     ```
   - авторизация запроса — секрет бота, значение передаётся отдельно
     (не хранить в коде бота в открытом виде).
3. Пользователю в чате бот показывает короткий токен вместо ссылки —
   пользователь вводит его в приложении FatVPN на экране онбординга.

## Миграция (без слома текущих продаж)

Обратная совместимость обязательна: пользователи, которые уже купили ключ
и не нажимали «сменить ключ», продолжают работать как раньше. Токен
выдаётся только в момент явного нажатия кнопки — старая выдача ссылок не
трогается.

## Что нужно от разработчика бота

- Адрес `POST /internal/tokens` и секрет — предоставит FatVPN-команда
  после готовности Дня 2/6.
- Подтверждение формата короткого токена и его длины.
- Тестирование только на `@testfatvpnnbot` перед переносом на боевой бот.
