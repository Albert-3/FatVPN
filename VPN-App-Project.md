на тёмно-синем фоне `#0B1622`. Акцентный цвет — мятно-зелёный из логотипа.

## 1. Краткое описание проекта

Разработка кросс-платформенного мобильного VPN-приложения (iOS + Android),
которое подключается к существующей инфраструктуре на базе **панели Remnawave**.
За основу (референс) взято приложение **Sota Connect** — минималистичный интерфейс,
одна кнопка подключения.

**Референс Sota:** приложение построено на **sing-box** (подтверждено пунктом
«Network stack: Mixed» в настройках — это дословная терминология sing-box).

## 3. Как работает система токенов (ключевая логика)

**Проблема заказчика:** непонятно, почему код Соты вставляется только в Соту.

**Объяснение:**
- Обычная sub-ссылка Remnawave (`https://panel.../sub/UUID`) — универсальная,
  вставляется в любой клиент.
- Sota выдаёт **короткий токен** (код), который расшифровывает **промежуточный
  сервер приложения (BFF)**, а не сама панель напрямую.
- По токену сервер находит нужную подписку в Remnawave и отдаёт приложению
  готовый конфиг. Поэтому код работает только в «своём» приложении.

**Что нужно доработать:** логику Telegram-бота — чтобы при продаже он выдавал
короткий токен и регистрировал ключ в BFF (вместо длинной ссылки).

> ⚠️ ВАЖНО (внутреннее): короткий токен Remnawave сам НЕ понимает.
> Нужен небольшой промежуточный сервер (backend/BFF) + БД, который хранит
> соответствие `токен → подписка`. Это входит в 10 дней — НЕ «просто доработка бота».

### Плавный переход (боевой бот не ломаем)

Заказчик боится, что переделка сломает текущие продажи. Договорённость:
1. **Сейчас** — боевой бот работает как есть, продажи идут по-старому. Не трогаем.
2. **В разработке** — вся логика токенов настраивается и тестируется на **тестовом
   боте** (заказчик его поднимет, подключит к своей панели, даст доступ).
3. **На релизе** — переносим изменения на боевой бот. Токены включаются вместе с
   приложением. Обратная совместимость: старая выдача может работать параллельно,
   уже купленные ключи не слетают.

### Миграция через кнопку «Сменить ключ» (подтверждено заказчиком)

В боте уже есть функция: выбираешь ключ → «поменять ключ» → бот выдаёт новый
актуальный ключ **с тем же сроком действия**.

**План миграции:** в новой версии эта кнопка выдаёт ключ уже в **новом формате
(короткий токен)** с сохранением текущего `expires_at`. Любой пользователь (новый
или уже купивший) жмёт «поменять ключ», получает токен и вставляет в приложение —
оплаченные дни сохраняются.

> Внутреннее: при доработке бота «подцепляемся» именно к кнопке «сменить ключ» —
> она должна генерировать короткий токен и регистрировать его в BFF (с текущим
> сроком из Remnawave). Логику покупки не трогаем — меняем только точку генерации ключа.

## 4. Архитектура

```
┌─────────────┐      ┌──────────────────────┐      ┌─────────────────┐
│  Мобильное  │◄────►│  BFF / API (.NET)     │◄────►│ Панель Remnawave│
│  приложение │ HTTPS│  - Auth / JWT         │      │  (Admin API)    │
│  (Flutter)  │      │  - Логика триала      │      └─────────────────┘
│             │      │  - Прокси серверов    │      ┌─────────────────┐
│ sing-box    │      │  - Токен → подписка   │◄────►│  Telegram-бот   │
│ (нативно)   │      └──────────┬────────────┘      │  + платежи      │
└─────────────┘                 ▼                    └─────────────────┘
                          ┌──────────┐
                          │ Postgres │ (устройства, триалы, токены)
                          └──────────┘
```

**Принцип:** приложение НИКОГДА не обращается к admin API Remnawave напрямую.
Все admin-креды хранит BFF. Приложение работает только со своим токеном.

## 5. Технологический стек (решение)

| Часть | Технология | Причина |
|---|---|---|
| Backend / BFF | **.NET (ASP.NET Core)** | Сильная сторона разработчика, отлично подходит |
| Мобильное приложение | **Flutter** | Есть готовые примеры VPN-клиентов sing-box, меньше риска |
| VPN-движок | **sing-box (Libbox)** | Покрывает Hysteria2, VLESS, gRPC, TCP; референс Sota на нём же |
| Туннель iOS | Нативно **Swift** (`NEPacketTunnelProvider`) | Обязательно, лимит памяти ~50 МБ |
| Туннель Android | Нативно **Kotlin** (`VpnService`) | Обязательно, foreground-сервис |
| БД | **PostgreSQL** | — |

**Почему НЕ .NET MAUI для приложения:** мало боевых VPN на MAUI, тяжело с
Network Extension на iOS (среда .NET + лимит памяти), капризный биндинг Go-библиотек.
Flutter — проверенный путь для sing-box/Xray клиентов.

### Протоколы (что умеет sing-box)
- **VLESS + Reality (TCP)** — вероятный основной протокол, устойчив к блокировкам
- **Hysteria2** — QUIC/UDP, быстрый на мобильных сетях
- **gRPC, xHTTP** — запасные транспорты
- ⚠️ **xHTTP** — поддержка в sing-box зависит от версии → СПАЙК на день 1

### Ключевые пакеты Flutter
- `dio` / `http` — запросы к backend
- `flutter_riverpod` или `bloc` — состояние
- `flutter_secure_storage` — хранение токена
- `device_info_plus` — Device ID для триала
- мост к sing-box — свой platform channel (референс: `flutter_singbox`/`flutter_v2ray`)

## 6. Логика триала (2–3 дня, безопасно)

Триал выдаётся **автоматически при первой установке** каждому новому пользователю
(без ввода кода). Наивный подход («device ID с клиента») ломается переустановкой.
Правильно:

- **iOS — DeviceCheck / App Attest:** 2 бита хранилища на устройство, переживают
  переустановку, контролируются серверами Apple. Бит = «триал использован».
- **Android — Play Integrity API:** BFF проверяет integrity-токен на сервере.

**Правила (всё на BFF):**
1. Приложение шлёт токен аттестации (App Attest / Play Integrity).
2. BFF проверяет токен на серверной стороне (не доверять клиенту).
3. Считает device-ключ, хеширует (солёный хеш, не сырой ID — приватность).
4. Если триал уже был → отказ. Иначе → Remnawave выдаёт подписку на 2–3 дня.
5. Срок истечения контролирует Remnawave, НЕ клиентские часы.

## 9. iOS без Mac — как тестировать и собирать

**Проблема:** разработчик на Windows 11. Собрать/подписать iOS-приложение можно
ТОЛЬКО на macOS.

**Решение (Mac покупать не обязательно):**
- Код пишем на Windows.
- Сборку iOS делает **Codemagic** (или GitHub Actions, macOS-раннер) — Mac в облаке.
- Готовую сборку заливаем в **TestFlight**.
- Тестируем на **реальном iPhone** (он есть у разработчика) через TestFlight.

**Важно:** VPN-туннель (Network Extension) нормально работает только на реальном
устройстве, не в симуляторе → физический iPhone нужен и полезен.

⚠️ Самая муторная часть iOS — code signing + entitlements для Network Extension
через облако. Заложить отдельное время.

### Окружение
- Windows (Android): Flutter SDK, Android Studio, VS Code + плагин Flutter, Git
- Mac/облако (iOS): Xcode, CocoaPods

## 12. Подводные камни сторов (VPN-приложения)

**Apple:**
- ⚠️ **Оплата через Telegram, без IAP** — риск реджекта по Guideline 3.1.1.
  Нужно решить заранее (позиционирование как клиент внешнего сервиса / демо-доступ).
- Guideline 5.4: VPN-приложение публикуется только от **организации** (не физлица).
- Privacy Nutrition Label: честно декларировать сбор данных; фингерпринтинг триала
  делать через DeviceCheck (санкционировано Apple).
- Лимит памяти Network Extension (~50 МБ).
- **Send logs** — в логах НЕ должно быть истории трафика/доменов.

**Google Play:**
- Политика VpnService: foreground-сервис + постоянное уведомление + явное согласие.
- Нельзя использовать VPN для манипуляции рекламой/сбора чужих данных.
- Декларация `FOREGROUND_SERVICE_SPECIAL_USE`.
- Data safety форма должна соответствовать реальности.

**Обе:**
- Обязательна хостинг-политика приватности.
- Ревьюеры должны иметь возможность протестировать — дать **демо-токен** (не прятать
  за Telegram, иначе реджект «не смогли проверить»).
